# Requirement Checklist

在设计或编码某个功能/模块之前，使用这个 Checklist。

## 目标

把功能想法整理成足够具体、可以设计和测试的需求。

## 检查项

### 业务目标

- [ ] 这个功能解决什么问题？
- [ ] 谁需要这个功能？
- [ ] 为什么现在需要它？

### 范围

- [ ] 包含什么？
- [ ] 不包含什么？
- [ ] 最小可用版本是什么？

### 使用者与权限

- [ ] 谁可以使用这个功能？
- [ ] 是否存在角色或权限差异？
- [ ] 每个角色可以查看或修改哪些数据？

### 功能行为

- [ ] 用户可以执行哪些操作？
- [ ] 需要哪些数据？
- [ ] 每个操作的预期结果是什么？
- [ ] 是否需要搜索、筛选、分页、导出、导入、通知或审批？

### 边界情况

- [ ] 数据缺失时怎么办？
- [ ] 输入无效时怎么办？
- [ ] 用户没有权限时怎么办？
- [ ] 外部服务失败时怎么办？
- [ ] 出现重复或过期数据时怎么办？

### 非功能需求

- [ ] 是否有性能要求？
- [ ] 是否有安全要求？
- [ ] 是否有审计/日志要求？
- [ ] 是否有兼容性约束？

### 验收标准

- [ ] 用户如何判断这个功能完成了？
- [ ] 哪些测试用例必须通过？
- [ ] 哪些内容需要人工验证？

## 输出

使用 `TEMPLATES/requirement.md` 生成需求文档。

如果重要信息缺失，先提出聚焦问题，再生成最终需求。
