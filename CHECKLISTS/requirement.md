# Requirement Checklist

在设计或编码某个功能/模块之前，使用这个 Checklist。

## 目标

把功能想法整理成足够具体、可以设计和测试的需求。

## 检查项

### 业务目标

- [ ] 这个功能解决什么问题？
- [ ] 谁需要这个功能？
- [ ] 为什么现在需要它？
- [ ] 如果不做这个功能，会有什么影响？

### 范围

- [ ] 包含什么？
- [ ] 不包含什么？
- [ ] 最小可用版本是什么？
- [ ] 有没有容易被误解为“顺便也要做”的内容？

### 使用者与权限

- [ ] 谁可以使用这个功能？
- [ ] 是否存在角色或权限差异？
- [ ] 每个角色可以查看或修改哪些数据？
- [ ] 数据归属是否清楚？例如属于个人、组织、团队、项目、店铺还是系统？

### 术语与命名

- [ ] 功能中涉及的核心名词是否已经统一？
- [ ] 是否存在同一个词代表多个概念的情况？
- [ ] 是否存在多个词代表同一个概念的情况？
- [ ] 是否需要把术语补充到项目需求摘要或当前需求文档中？

### 功能行为

- [ ] 用户可以执行哪些操作？
- [ ] 需要哪些数据？
- [ ] 每个操作的预期结果是什么？
- [ ] 是否需要搜索、筛选、分页、导出、导入、通知或审批？

### 边界情况

- [ ] 数据缺失时怎么办？
- [ ] 输入无效时怎么办？
- [ ] 用户没有权限时怎么办？
- [ ] 外部服务失败时怎么办？
- [ ] 出现重复或过期数据时怎么办？
- [ ] 用户取消、撤回、删除或恢复时怎么办？

### 非功能需求

- [ ] 是否有性能要求？
- [ ] 是否有安全要求？
- [ ] 是否有审计/日志要求？
- [ ] 是否有兼容性约束？

### 验收标准

- [ ] 用户如何判断这个功能完成了？
- [ ] 哪些测试用例必须通过？
- [ ] 哪些内容需要人工验证？
- [ ] 有没有明确的“不算完成”的情况？

## 拷问方式

当需求仍然模糊时，优先追问这些问题：

- 这个功能的第一版到底要解决哪一个最核心的问题？
- 哪些用户能做，哪些用户不能做？
- 数据属于谁？谁能看到？谁能修改？
- 正常流程之外，失败、重复、无权限、数据为空时怎么办？
- 什么情况可以明确验收通过？

追问结束后，把已确认的术语、范围、权限和验收标准写入需求文档，不要只留在聊天记录里。

## 输出

使用 `TEMPLATES/requirement.md` 生成需求文档。

如果重要信息缺失，先提出聚焦问题，再生成最终需求。
