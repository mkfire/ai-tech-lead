# Task Planning Checklist

当设计方案已经被用户确认，并且即将进入非轻量编码工作之前，使用这个 Checklist。

## 目标

生成一份 `docs/tasks/<feature-name>.md`，把设计拆成可执行、可验证、可 Review 的小任务。

任务文档只负责执行拆分和进度记录，不替代需求文档、设计文档或架构文档。

## 检查项

### 输入确认

- [ ] 需求文档是否存在，或已明确记录用户选择跳过需求？
- [ ] 设计文档是否已经输出并被用户确认？
- [ ] 是否已读取 `docs/architecture.md`，如果该文件存在？
- [ ] 是否已确认本次任务对应的功能或模块名称？

### 文件位置与命名

- [ ] 任务文档是否放在 `docs/tasks/`？
- [ ] 文件名是否能表达模块或功能，例如 `<module>-<feature>.md`？
- [ ] 是否避免使用过于笼统的文件名，例如 `task.md`、`todo.md`、`plan.md`？

### 任务粒度

- [ ] 是否优先拆成 2～5 个小任务？
- [ ] 每个任务是否能独立实现、验证和 Review？
- [ ] 是否避免把任务拆得过细，例如“写 import”“创建空文件”？
- [ ] 是否避免把任务拆得过大，例如“实现整个系统”？
- [ ] 如果任务超过 5 个，是否说明为什么不能继续合并或分阶段？

### 任务内容

每个任务都应包含：

- [ ] 文件范围：默认应该修改哪些文件或目录。
- [ ] 产出：这个任务完成后会得到什么。
- [ ] 验证方式：运行什么测试、命令，或执行什么人工验证步骤。
- [ ] 状态：todo / doing / done / blocked。
- [ ] 依赖：是否依赖前置任务。

### 范围控制

- [ ] 任务是否只覆盖当前已确认设计？
- [ ] 是否避免加入设计中没有要求的新功能？
- [ ] 是否避免把重构、优化、顺手清理混入业务任务？
- [ ] 如果确实需要重构，是否单独列出原因、文件范围和验证方式？

### 执行顺序

- [ ] 是否明确第一个要做的任务？
- [ ] 是否说明哪些任务必须串行？
- [ ] 如果任务可以并行，是否说明并行边界？
- [ ] 是否说明阻塞条件和下一步？

### 验证记录

- [ ] 是否预留验证记录位置？
- [ ] 是否能记录每个任务执行后的验证命令或人工验证步骤？
- [ ] 是否能记录验证结果和剩余风险？

## 输出

使用 `TEMPLATES/task.md` 生成任务文档：

```text
docs/tasks/<module>-<feature>.md
```

生成后，先让用户确认任务拆分；用户确认后再进入 Coding。
