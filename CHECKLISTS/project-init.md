# Project Init Checklist

在 Discovery 完成、项目需求摘要已经足够清楚、可以选择项目结构时，使用这个 Checklist。

## 目标

根据真实需求初始化一个传统、清晰、容易理解、可维护的项目结构。

## 检查项

### 项目形态

- [ ] 这是简单脚本、CLI 工具、前端应用、后端服务、全栈应用、管理后台、浏览器插件、移动端应用、库 / SDK、数据处理服务、AI Agent、Monorepo，还是多服务系统？
- [ ] 是否需要拆分 `backend/` 和 `frontend/`？
- [ ] 是否需要 `apps/` 和 `packages/`？
- [ ] 现在是否需要基础设施或部署配置？
- [ ] 判断依据是否来自需求，而不是来自默认偏好？

### 架构形态选择

- [ ] 是否选择了能支撑 V1 的最简单架构？
- [ ] 是否说明为什么当前不需要更复杂结构？
- [ ] 是否存在前后端分离信号：独立前端、后端 API、移动端/插件/第三方共用接口？
- [ ] 是否存在 Monorepo 信号：多个应用需要共享类型、SDK、组件或业务规则？
- [ ] 是否存在队列/任务系统信号：异步同步、大批量导入、定时任务、外部 API 不稳定、重试或补偿？
- [ ] 是否存在多服务信号：独立团队维护、部署/扩容需求不同、强隔离要求？

### 结构推导拷问

当项目结构存在不确定点时，先给出初步判断，再拷问用户。不要直接问用户“你想怎么设计目录”。

- [ ] 是否先给出项目类型、架构形态和核心模块的初步判断？
- [ ] 是否说明判断依据？
- [ ] 是否列出关键假设？
- [ ] 是否只追问会影响架构形态、模块边界、目录结构、数据归属、部署方式或扩展方式的问题？
- [ ] 是否避免把所有架构问题一次性丢给用户？
- [ ] 用户确认后，是否把结论写入 `docs/architecture.md`？

### 代码结构与模块边界

- [ ] 是否识别出 V1 的核心业务能力？
- [ ] 是否按业务能力划分模块，而不是只按全局技术层划分？
- [ ] 每个模块是否说明职责？
- [ ] 每个模块是否说明拥有的数据？
- [ ] 每个模块是否说明对外接口？
- [ ] 每个模块是否说明不应该负责的事情？
- [ ] 是否给出推荐目录结构？
- [ ] 是否说明后续新增功能应该放在哪里？

### 模块依赖规则

- [ ] 是否说明模块之间如何协作？
- [ ] 是否限制跨模块直接访问数据？
- [ ] 是否说明公共目录 `shared/`、`common/`、`lib/` 只能放稳定通用能力？
- [ ] 是否禁止把业务逻辑随手放进公共目录？
- [ ] 是否说明 Codex 开发单个功能时默认只修改相关模块？

### 文档

- [ ] 根据 `AGENTS.template.md` 创建 `AGENTS.md`。
- [ ] 创建 `docs/project-brief.md`。
- [ ] 创建 `docs/architecture.md`。
- [ ] 创建 `docs/requirements/`。
- [ ] 创建 `docs/design/`。
- [ ] 创建 `docs/tasks/`。
- [ ] 创建 `docs/decisions/`。

### 架构基线

- [ ] 记录项目类型。
- [ ] 记录技术栈，如果已经选择。
- [ ] 记录代码结构与模块边界。
- [ ] 记录模块依赖规则。
- [ ] 记录数据/存储方案，如果已知。
- [ ] 记录部署假设，如果已知。

### 范围控制

- [ ] 不要在需求不足时生成复杂结构。
- [ ] 除非明确必要，不要引入微服务。
- [ ] 除非需求明确要求，不要引入队列、缓存、Kubernetes 或重型基础设施。
- [ ] 优先选择能支撑第一版的最简单结构。

## 推荐默认结构

```text
project/
├── AGENTS.md
├── docs/
│   ├── project-brief.md
│   ├── architecture.md
│   ├── requirements/
│   ├── design/
│   ├── tasks/
│   └── decisions/
└── src/
```

## 更复杂的全栈结构

仅在确实需要时使用：

```text
project/
├── AGENTS.md
├── docs/
│   ├── project-brief.md
│   ├── architecture.md
│   ├── requirements/
│   ├── design/
│   ├── tasks/
│   └── decisions/
├── backend/
├── frontend/
└── infra/
```

## 输出

初始化完成后，清楚报告：

- 创建了哪些文件和目录
- 为什么选择这个结构
- 为什么当前不需要更复杂结构
- 核心模块是什么，以及每个模块负责什么
- Codex 后续开发时默认应该修改哪些目录
- 结构推导中还有哪些假设或风险
- 下一步建议做什么
