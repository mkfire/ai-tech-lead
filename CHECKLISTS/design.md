# Design Checklist

当需求已经清楚，并且即将进行非简单编码工作之前，使用这个 Checklist。

## 目标

输出一份可 Review 的设计方案，说明如何安全实现需求。

设计内容应按复杂度缩放：简单功能可以短设计；复杂功能才展开模块、接口、数据模型、权限、异常流程和测试策略。

## 检查项

### 需求对齐

- [ ] 设计是否直接满足需求？
- [ ] 是否覆盖所有验收标准？
- [ ] 已排除的内容是否仍然排除？
- [ ] 设计是否引入了需求中没有要求的新行为？

### 设计确认方式

- [ ] 如果设计复杂，是否先分段确认，而不是一次性输出过长全文？
- [ ] 是否已确认范围和目标？
- [ ] 是否已确认模块边界？
- [ ] 是否已确认数据、API、权限或集成方案中的关键部分？
- [ ] 是否已确认流程、异常、测试和风险？
- [ ] 用户确认设计前，是否避免写代码、生成脚手架、改文件或执行实现动作？

### 方案判断

- [ ] 如果存在真实取舍，是否给出 2～3 个方案、主要差异和推荐理由？
- [ ] 如果没有真实取舍，是否避免为了形式编造备选方案？
- [ ] 推荐方案是否是满足 V1 的最简单可行方案？
- [ ] 是否说明哪些内容暂不做？

### 模块边界

- [ ] 哪个模块负责这个功能？
- [ ] 可能影响哪些文件或目录？
- [ ] 模块边界是否清晰？
- [ ] 是否需要跨模块调用？是否合理？
- [ ] 这个边界以后是否容易被误解或推翻？
- [ ] 每个新增或修改的单元是否职责单一、接口清楚、依赖明确？

### 接口

- [ ] API、命令、函数或 UI 事件是否明确？
- [ ] 输入和输出是否清楚？
- [ ] 错误响应是否清楚？
- [ ] 是否需要 DTO、类型或契约？
- [ ] 命名是否和项目已确认术语一致？

### 数据模型

- [ ] 会创建、读取、更新或删除哪些数据？
- [ ] 是否需要修改 Schema？
- [ ] 是否需要 Migration？
- [ ] 是否需要索引或约束？
- [ ] 数据归属、生命周期和删除/恢复规则是否清楚？

### 流程

- [ ] 主要成功路径是什么？
- [ ] 失败路径有哪些？
- [ ] 权限检查在哪里发生？
- [ ] 是否涉及外部服务调用？
- [ ] 是否有并发、重复提交、幂等或补偿问题？

### 视觉化判断

- [ ] 这个设计是否真的需要图示才能更清楚？
- [ ] 如果是 UI 布局、流程、架构关系或实体关系，是否可以用图示、流程图、架构图或 Mermaid 简化理解？
- [ ] 如果只是需求、范围或技术取舍，是否保持文字说明，避免不必要视觉化？

### 风险

- [ ] 什么可能破坏现有行为？
- [ ] 什么可能造成安全或权限问题？
- [ ] 哪些部分仍然不确定？
- [ ] 实现前或实现中应该验证什么？
- [ ] 如果用户要求直接 Coding，最小安全实现范围是什么？

### ADR 判断

只有当一个决策同时满足以下条件时，才建议创建 ADR：

- [ ] 难以回滚：后续改变成本明显较高。
- [ ] 没有上下文会让人困惑：未来维护者可能会问“为什么这样做”。
- [ ] 存在真实取舍：有可行替代方案，但基于原因选择了当前方案。

如果只是显而易见的小选择，或很容易改回，不要创建 ADR。

### 测试策略

- [ ] 需要哪些单元测试？
- [ ] 需要哪些集成测试？
- [ ] 需要哪些人工验证？
- [ ] 应该运行哪些现有测试？

### 任务拆分

- [ ] 实现是否可以拆成小任务？
- [ ] 是否有任务可以安全并行？
- [ ] 应该先做哪个任务？

## 设计拷问

设计完成后，先拷问设计而不是马上编码：

- 这个方案有没有过度设计？
- 有没有更简单的方案满足 V1？
- 哪个边界最容易被 Codex 写错？
- 哪个权限或数据归属最容易出漏洞？
- 如果只做最小安全版本，应该砍掉什么？

把拷问后确认的关键结论写回设计文档；只有真正重要的、难以回滚的取舍才写入 ADR。

## 设计自检

设计文档输出后，先自检并修正明显问题：

- [ ] 是否存在 TBD、TODO、空章节或未完成占位？
- [ ] 是否存在前后矛盾？
- [ ] 是否超出当前需求或模块范围？
- [ ] 是否存在容易被 Codex 理解错的歧义？
- [ ] 是否能支持拆成清晰的小任务进入 Coding？

## 输出

使用 `TEMPLATES/design.md` 生成设计文档。

除非用户明确要求跳过设计，否则在用户确认设计前不要开始编码、生成脚手架、改文件或执行实现动作。
