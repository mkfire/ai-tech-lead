# Design Checklist

当需求已经清楚，并且即将进行非简单编码工作之前，使用这个 Checklist。

## 目标

输出一份可 Review 的设计方案，说明如何安全实现需求。

## 检查项

### 需求对齐

- [ ] 设计是否直接满足需求？
- [ ] 是否覆盖所有验收标准？
- [ ] 已排除的内容是否仍然排除？

### 模块边界

- [ ] 哪个模块负责这个功能？
- [ ] 可能影响哪些文件或目录？
- [ ] 模块边界是否清晰？
- [ ] 是否需要跨模块调用？是否合理？

### 接口

- [ ] API、命令、函数或 UI 事件是否明确？
- [ ] 输入和输出是否清楚？
- [ ] 错误响应是否清楚？
- [ ] 是否需要 DTO、类型或契约？

### 数据模型

- [ ] 会创建、读取、更新或删除哪些数据？
- [ ] 是否需要修改 Schema？
- [ ] 是否需要 Migration？
- [ ] 是否需要索引或约束？

### 流程

- [ ] 主要成功路径是什么？
- [ ] 失败路径有哪些？
- [ ] 权限检查在哪里发生？
- [ ] 是否涉及外部服务调用？

### 风险

- [ ] 什么可能破坏现有行为？
- [ ] 什么可能造成安全或权限问题？
- [ ] 哪些部分仍然不确定？
- [ ] 实现前或实现中应该验证什么？

### 测试策略

- [ ] 需要哪些单元测试？
- [ ] 需要哪些集成测试？
- [ ] 需要哪些人工验证？
- [ ] 应该运行哪些现有测试？

### 任务拆分

- [ ] 实现是否可以拆成小任务？
- [ ] 是否有任务可以安全并行？
- [ ] 应该先做哪个任务？

## 输出

使用 `TEMPLATES/design.md` 生成设计文档。

除非用户明确要求跳过设计，否则在用户确认设计前不要开始编码。
