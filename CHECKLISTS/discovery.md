# Discovery Checklist

当用户只有模糊项目想法，或要求启动一个新项目时，使用这个 Checklist。

## 目标

把模糊想法整理成简洁的项目需求摘要，用于后续项目初始化和技术方案设计。

## 检查项

### 问题与目标

- [ ] 这个项目解决什么问题？
- [ ] 谁遇到了这个问题？
- [ ] 第一版要达到什么结果？
- [ ] 什么情况代表这个项目成功了？

### 用户与角色

- [ ] 谁会使用这个系统？
- [ ] 是否存在不同用户角色？
- [ ] 是否需要登录？
- [ ] 是否需要权限或访问控制？

### 第一版范围

- [ ] 第一版必须做的 3 个功能是什么？
- [ ] V1 明确不做什么？
- [ ] 哪个功能应该最先开发？
- [ ] 哪些功能可以后置？

### 核心流程

- [ ] 主要用户流程是什么？
- [ ] 会创建、导入、编辑或查看哪些数据？
- [ ] 是否存在审批、Review 或审计步骤？
- [ ] 是否需要定时任务或后台任务？

### 数据与集成

- [ ] 数据从哪里来？
- [ ] 是否需要对接第三方服务？
- [ ] 是否需要文件上传、支付、邮件、消息、采集、浏览器插件、移动端或管理后台？
- [ ] 是否需要报表或导出？

### 非功能需求

- [ ] 是否有性能要求？
- [ ] 是否有安全或隐私要求？
- [ ] 是否有部署限制？
- [ ] 这是原型、内部工具、SaaS、生产系统还是企业系统？

### 风险与未知项

- [ ] 当前还有什么不清楚？
- [ ] 哪些决策后续可能导致大返工？
- [ ] 编码前应该先验证什么？

## 输出

完成 Discovery 后，使用 `TEMPLATES/project-brief.md` 生成 `docs/project-brief.md`。

在项目需求摘要足够清楚之前，不要初始化项目骨架。
