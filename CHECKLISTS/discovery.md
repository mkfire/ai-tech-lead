# Discovery Checklist

当用户只有模糊项目想法，或要求启动一个新项目时，使用这个 Checklist。

## 目标

把模糊想法整理成简洁的项目需求摘要，用于后续项目初始化和技术方案设计。

## 检查项

### 问题与目标

- [ ] 这个项目解决什么问题？
- [ ] 谁遇到了这个问题？
- [ ] 第一版要达到什么结果？
- [ ] 什么情况代表这个项目成功了？

### 用户与角色

- [ ] 谁会使用这个系统？
- [ ] 是否存在不同用户角色？
- [ ] 是否需要登录？
- [ ] 是否需要权限或访问控制？
- [ ] “用户”“客户”“账号”“成员”“管理员”“操作者”等词是否已经区分清楚？

### 第一版范围

- [ ] 第一版必须做的 3 个功能是什么？
- [ ] V1 明确不做什么？
- [ ] 哪个功能应该最先开发？
- [ ] 哪些功能可以后置？
- [ ] 有没有用户说起来想要、但其实会拖慢 V1 的功能？

### 核心流程

- [ ] 主要用户流程是什么？
- [ ] 会创建、导入、编辑或查看哪些数据？
- [ ] 是否存在审批、Review 或审计步骤？
- [ ] 是否需要定时任务或后台任务？
- [ ] 流程失败、取消、重复提交或数据冲突时怎么办？

### 数据与集成

- [ ] 数据从哪里来？
- [ ] 是否需要对接第三方服务？
- [ ] 是否需要文件上传、支付、邮件、消息、采集、浏览器插件、移动端或管理后台？
- [ ] 是否需要报表或导出？
- [ ] 核心数据归属是否清楚？例如数据属于用户、组织、资源、项目、工作区还是系统？

### 术语与命名

- [ ] 是否出现模糊或重复概念？
- [ ] 是否已经为核心业务概念选择统一叫法？
- [ ] 是否明确哪些词不要混用？
- [ ] 是否需要把术语写入 `docs/project-brief.md` 的“术语与命名”部分？

### 非功能需求

- [ ] 是否有性能要求？
- [ ] 是否有安全或隐私要求？
- [ ] 是否有部署限制？
- [ ] 这是原型、内部工具、SaaS、生产系统还是企业系统？

### 风险与未知项

- [ ] 当前还有什么不清楚？
- [ ] 哪些决策后续可能导致大返工？
- [ ] 编码前应该先验证什么？
- [ ] 有没有需要现在记录的关键取舍？如果没有明显取舍，不要写 ADR。

## 拷问方式

当用户回答仍然模糊时，进入拷问模式：

- 优先追问会影响项目结构的内容：用户角色、权限、数据来源、核心模块、V1 范围。
- 不要一次性问太多问题，先问最可能导致返工的 3～6 个问题。
- 对容易混淆的术语要直接指出，并要求用户选择统一叫法。
- 每轮追问后，简短总结已经确认的内容和仍然缺失的内容。

## 输出

完成 Discovery 后，使用 `TEMPLATES/project-brief.md` 生成 `docs/project-brief.md`。

在项目需求摘要足够清楚之前，不要初始化项目骨架。
