# Coding Checklist

当需求、设计和任务拆分已经准备好，并且用户已经确认可以实现时，使用这个 Checklist。

## 目标

按 `docs/tasks/` 中的任务，用最小、安全、可 Review 的改动实现目标功能。

## 检查项

### 编码前

- [ ] 需求已经存在，或已明确记录用户选择跳过需求。
- [ ] 对于非简单修改，设计已经存在，或已明确记录用户选择跳过设计。
- [ ] 对于非轻量功能，已读取对应 `docs/tasks/<feature-name>.md`，或已明确记录用户选择跳过任务拆分。
- [ ] 已读取 `docs/architecture.md`，如果该文件存在。
- [ ] 已读取对应需求文档和设计文档，如果它们存在。
- [ ] 已查看相关目录、相关代码和现有实现风格。
- [ ] 已识别本次任务的文件范围、产出和验证方式。
- [ ] 已确认本次修改属于哪个模块，以及默认应该修改哪些目录。
- [ ] 已明确测试或验证计划。

### 范围控制

- [ ] 只修改当前任务需要的文件。
- [ ] 不做无关重构。
- [ ] 除非设计或任务文档要求，不重命名或重组模块。
- [ ] 除非设计或任务文档要求，不修改公开接口。
- [ ] 只改对当前目标有直接帮助的结构问题，不为了“看起来更好”重构。
- [ ] 如果发现任务拆分不合理，先暂停并更新任务文档或说明风险，不要自行扩大范围。

### 实现

- [ ] 遵循现有项目风格。
- [ ] 遵循现有目录结构、命名风格和模块边界。
- [ ] 把业务逻辑放在合适的层。
- [ ] 校验输入并处理错误。
- [ ] 遵守权限和安全规则。
- [ ] 保持代码可读、可维护。

### 测试与验证

- [ ] 在可行时新增或更新测试。
- [ ] 运行当前任务对应的测试、Lint、类型检查或构建命令。
- [ ] 如果无法运行测试，说明原因并提供人工验证步骤。

### 完成总结

报告：

- 完成了 `docs/tasks/` 中的哪个任务
- 修改了哪些文件
- 改变了什么行为
- 运行了哪些测试或验证
- 是否需要更新任务状态
- 剩余风险或后续工作

## 输出

只有当这个 Checklist 满足，或用户明确选择带风险继续时，才开始实现代码。
