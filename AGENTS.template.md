# AGENTS.md

本项目使用 AI Tech Lead 工作流辅助开发。

## 项目规则

不要把 AI 当成直接写代码的工具。应该把 AI 当成工程协作者，必须按“澄清需求 → 输出设计 → 实现代码 → Review 验收”的顺序推进。

当用户提出新项目、功能开发、需求整理、设计方案、代码实现或 Review 请求时，优先使用 AI Tech Lead Skill。

## 默认开发流程

对于非简单修改，遵循以下流程：

1. 澄清需求。
2. 输出或更新设计方案。
3. 当设计会影响行为、架构、API、数据模型或权限时，等待用户确认。
4. 用最小改动实现。
5. 运行可用的验证步骤。
6. 总结修改文件、测试结果和剩余风险。

简单修改可以轻量处理，例如 typo、简单文案、一行配置、小范围样式调整或无行为影响的注释整理。即使轻量处理，也要保持最小改动，并说明修改内容和验证结果。

## 文档结构

使用传统项目文档结构：

```text
docs/
├── project-brief.md
├── architecture.md
├── requirements/
├── design/
└── decisions/
```

## 需求规则

- 需求是长期维护的事实来源。
- 如果需求不清楚，先提问，不要直接写代码。
- 需求应包含使用者、范围、行为、权限、边界情况和验收标准。

## 本项目架构规则

项目初始化后，应在 `docs/architecture.md` 中记录本项目的具体架构基线，包括：

- 项目类型与架构形态。
- 核心模块、模块职责和数据归属。
- 推荐目录结构。
- 模块之间的调用方式和依赖边界。
- `shared/`、`common/`、`lib/` 等公共目录的使用规则。

后续开发应遵循 `docs/architecture.md`。如果实际代码结构与架构文档不一致，先判断是文档过期还是实现偏离；涉及模块边界、数据归属或公共目录规则变化时，先更新架构文档或说明原因。

## 设计规则

- 复杂修改必须先有设计方案。
- 设计应说明模块、接口、数据模型、流程、风险和测试策略。
- 设计内容按复杂度缩放：简单功能可以短设计，复杂功能才展开所有细节。
- 复杂设计应分段确认，优先确认范围和目标、模块边界、数据/API/权限、流程异常、测试风险。
- 非轻量任务在设计确认前，不要写代码、生成脚手架、改文件或执行实现动作。
- 如果设计与需求冲突，以需求为准。
- 如果设计改变模块边界、数据归属、权限模型或公共目录，必须明确说明原因和影响范围。
- 只有当图示比文字更清楚时，才使用图示、流程图、架构图或 Mermaid；需求、范围和技术取舍默认用文字说明。

## 编码规则

- 做最小安全改动。
- 默认只修改当前功能所属模块。
- 如果需要修改其他模块，必须先说明原因和影响范围。
- 如果需要修改 `shared/`、`common/`、`lib/` 等公共目录，必须说明为什么不能放在业务模块内。
- 不修改无关模块。
- 不做顺手重构。
- 不把业务逻辑塞进公共工具目录。
- 除非需求明确要求，否则保持现有行为不变。
- 在可行时补充或更新测试。

## 高风险操作

以下操作必须先获得用户确认：

- 删除核心配置文件。
- 破坏性数据库 Schema 变更。
- 影响生产环境、真实账号、外部服务或真实数据的操作。
- 推送到远程仓库、发布上线或执行不可逆部署操作。

## Review 规则

完成前检查：

- 是否满足需求
- 是否符合设计
- 是否控制了范围
- 是否遵守模块边界
- 是否出现不必要的跨模块修改
- 权限和安全是否正确
- 边界情况是否覆盖
- 是否完成测试或验证
- 是否需要更新文档

当修改需求、架构、API、权限、数据模型、部署方式或核心业务流程时，应同步更新对应文档；如果不需要更新，应在总结中说明原因。

## 用户覆盖规则

用户可以选择跳过某个推荐阶段。

如果用户选择跳过，必须清楚说明被跳过的阶段、对应风险，以及最安全的最小下一步。
